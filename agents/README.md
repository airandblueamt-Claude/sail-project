# `agents/`

Externalised agent definitions for SAIL. The Python runtime
(`routes/assistant.py`) loads these files at request time so the
agent's behaviour can be tuned by editing prose, not code.

## Conventions

- One markdown file per agent. Today there's one: `sail_helper.md`.
- The file is the **system prompt** the model sees, verbatim, with
  one substitution: the placeholder `{snapshot}` is replaced with a
  live DB snippet generated by Python (counts, who the user is).
- Edits take effect on the **next chat message**. The file is read
  on every request (cheap — it's ~5 KB) so there's no server
  restart needed. If you want to disable hot-reload during a noisy
  edit, set `SAIL_AGENT_HOTRELOAD=0` in the env.
- Tool *definitions* (the `find_tickets`, `get_ticket`, … functions)
  stay in `routes/assistant_tools.py` because each one is a real
  Python function. The markdown describes **when to call them**;
  the Python describes **how they execute**.

## Adding a new tool

1. Implement the function and add it to `TOOLS` + `TOOL_SPECS` in
   `routes/assistant_tools.py`.
2. Add a line under "Tool playbook" in `sail_helper.md` explaining
   when the model should call it.

## Changing the agent's tone

Edit the "How to answer" section of `sail_helper.md`. Common knobs:

- Length cap ("1–3 sentences" → "1 paragraph max").
- Style ("Be terse" → "Be warm and conversational").
- Refusal policy ("scoped to SAIL only" → "answer anything").

## Why markdown instead of YAML / JSON

The system prompt is essentially prose written for an LLM. A markdown
file:
- Reads naturally — anyone on the team can audit "what does the AI
  know" without parsing YAML.
- Diffs well in git — reviewers see the prose changes directly.
- Is what the model literally sees, with no transformation. There's
  no risk of the YAML-to-prompt conversion drifting from intent.
